## 新增需求

### 需求：Flutter 端玻璃表面容器
Flutter 移动端所有卡片、对话框和控制栏表面必须使用 `BackdropFilter` 配合 `ImageFilter.blur()` 对背景层进行模糊处理，结合半透明渐变填充（`Colors.white.withOpacity(0.05)` 至 `0.1`）和微妙的白色边框（`Colors.white.withOpacity(0.1)`）。

#### 场景：首页操作按钮使用玻璃效果
- **当** 首页渲染投屏或接收操作按钮时
- **则** 按钮容器必须使用 `BackdropFilter` 模糊背后内容，配合半透明渐变填充和 `Colors.white10` 边框

#### 场景：投屏控制栏使用玻璃效果
- **当** 投屏界面渲染底部控制栏时
- **则** 控制栏必须使用背景模糊、半透明深色填充和圆角边框，悬浮于画面预览之上

#### 场景：付费墙对话框使用玻璃效果
- **当** 付费墙对话框显示时
- **则** 对话框背景必须使用背景模糊和半透明填充，配合微妙的青色强调边框

### 需求：Web 端玻璃表面容器
Vue Web 端所有面板、模态框和控制栏必须使用 Tailwind `backdrop-blur-xl`（或更强）结合 `bg-white/5`（或更浅）和 `border-white/10` 来实现磨砂玻璃外观。

#### 场景：首页卡片使用玻璃效果
- **当** 首页渲染时
- **则** 操作按钮和信息卡片必须具有 `backdrop-blur-xl bg-white/5 border-white/10` 样式

#### 场景：接收端控制栏使用玻璃效果
- **当** 接收端活跃视图渲染底部控制栏时
- **则** 控制栏必须使用 `backdrop-blur-2xl bg-black/40 border-white/10` 样式

#### 场景：Toast 通知使用玻璃效果
- **当** Toast 通知出现时
- **则** Toast 必须使用 `backdrop-blur-md` 和半透明背景

### 需求：统一青色强调色调色板
应用必须在 Flutter 移动端和 Vue Web 端统一使用青色/蓝绿色调（`#06B6D4` / cyan-500 等效值），替换移动端靛蓝主色（#6366F1）和 Web 端琥珀色（#f59b0b）强调色。

#### 场景：移动端主色为青色
- **当** 任何 Flutter 组件引用 `kPrimaryColor` 时
- **则** 解析后的颜色必须为青色（#06B6D4）而非靛蓝（#6366F1）

#### 场景：Web 端强调色为青色
- **当** 任何 Vue 组件使用强调色文字、按钮或边框时
- **则** Tailwind 类名必须引用 cyan（cyan-400/500）而非 amber（amber-500）

### 需求：分层模糊深度体系
设计体系必须定义三级模糊层级：轻微（sigma=4，用于卡片）、中等（sigma=8，用于对话框/模态框）和重度（sigma=16，用于全屏遮罩）。每层必须根据 UI 元素的 Z 轴突出程度一致应用。

#### 场景：卡片使用轻微模糊
- **当** 信源选择卡片或投屏码数字渲染时
- **则** 必须使用 sigma=4 模糊（Flutter）或 `backdrop-blur-sm`（Web）

#### 场景：对话框使用中等模糊
- **当** 付费墙对话框或信息模态框渲染时
- **则** 必须使用 sigma=8 模糊（Flutter）或 `backdrop-blur-xl`（Web）

#### 场景：遮罩使用重度模糊
- **当** 加载遮罩或全屏投屏预览渲染时
- **则** 必须使用 sigma=16 模糊（Flutter）或 `backdrop-blur-2xl`（Web）

### 需求：渐变边框高光
所有玻璃容器必须具有微妙的渐变边框高光（左上较亮、右下较暗），以模拟玻璃边缘的光线反射。Flutter 端通过 `LinearGradient` 边框实现，Web 端通过 `border-t-white/20 border-l-white/15` 结合 `border-b-white/5 border-r-white/5` 实现。

#### 场景：卡片具有渐变边框
- **当** 玻璃卡片渲染时
- **则** 上边框和左边框必须略亮于下边框和右边框

### 需求：无功能回归
所有玻璃效果视觉变更不得改变任何现有功能行为：媒体采集、WebRTC 信令、订阅逻辑、试用计时器、投屏码生成和导航流程必须与变更前保持一致。

#### 场景：投屏流程不变
- **当** 用户在玻璃 UI 变更后发起投屏时
- **则** 屏幕采集、投屏码生成、PeerJS 连接、试用计时器启动和画面传输必须与之前完全一致

#### 场景：接收流程不变
- **当** 用户在玻璃 UI 变更后加入画面时
- **则** 投屏码输入、PeerJS 握手、视频渲染和画中画控制必须与之前完全一致
