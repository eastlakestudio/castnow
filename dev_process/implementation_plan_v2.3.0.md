# 实施方案 - iOS App 媒体功能升级 (v2.3.0)

本方案旨在升级 `mobile_pro` Flutter 应用，使其具备与 Web 端对等的媒体传输能力：支持同时广播屏幕、摄像头和麦克风流。

## 用户审核事项

> [!IMPORTANT]
> **iOS 并发采集**：在 iOS 上，同时采集屏幕和摄像头是通过 Broadcast Extension 和 `getUserMedia` 实现的。我将确保逻辑能够将这些轨道正确合并到单个 PeerJS 呼叫中，保持与 Web 接收端的一致性。

## 拟议变更

### [mobile_pro] [main.dart](file:///Users/minghualiu/personal/EastlakeStudio/castnow/apps/mobile_pro/lib/main.dart)

#### 1. 状态与默认值
- **默认静音**：初始化 `_isMuted = true`。根据要求，确保麦克风在启动时处于静音状态。
- **源选择**：将 `_shareScreen`、`_shareCamera` 和 `_shareMic` 的初始状态均设为 `true`，与 Web 端优化后的体验保持一致。

#### 2. 广播逻辑重构
- **并发采集**：重构 `_startBroadcast` 函数。如果同时选择了屏幕和摄像头：
    - 优先启动屏幕共享。
    - 接着启动摄像头采集。
    - 将所有视频轨道添加到最终的 `MediaStream` 中。
- **麦克风处理**：即使视频源已激活，也要确保添加音频轨道。
- **校验逻辑**：强制要求在启动前至少选择一个视频源（屏幕或摄像头）。

#### 3. UI 细节优化
- **交互切换**：确保“选择源”界面清晰支持多选。
- **状态显示**：更新“SHARING” / “ON AIR”状态逻辑，准确反映当前激活的媒体源。
- **镜像处理**：确保摄像头预览仅在作为主要源（前置摄像头）时进行镜像。

## 验证计划

### 手动验证
1. **源选择验证**：打开应用，确认屏幕、摄像头和麦克风默认勾选。取消勾选屏幕和摄像头，点击启动应弹出警告。
2. **双流测试**：启动同时包含屏幕和摄像头的广播。在 Web 接收端验证是否收到双流，并能正常使用画中画或并列模式切换。
3. **默认静音验证**：启动后确认麦克风图标为红色（静音），手动点击后可解除静音。
